## Water Year Convention

All simulation periods use the **SFWMD water year**: May 1 – April 30, named by
the end year (e.g., WY2026 = May 1, 2025 – April 30, 2026). The first rainfall of
the wet season typically arrives around May 15.

## 1. Rainfall — `DailyRainFilled_cm_1999-9-1_2026-4-30.csv`

### Original file
`data/Rain/DailyRainFilled_cm_1999-9-1_2016-12-31.csv`
17 stations, daily cm/day, gap-filled. Columns: `BK_cm_day`, `BA_cm_day`, …

### Data source
Internal TEF04 MySQL database (`hydrology.measurement`, `datatype='rain'`).
17 ENP/Florida Bay rain gauge stations identified by two-letter codes matching
column headers (e.g., BK, BA, MK).

### Unit conversion
Raw DB values are in **in/hr** at sub-daily resolution. Aggregated in SQL as:
`SUM(measurement_value)` per day per station. Mode of daily reading count = 24
(hourly intervals), confirming `hours_per_interval = 1.0`.
Conversion: `daily_cm = sum_in_hr * 1.0 hr * 2.54 cm/in`

### Extension period
`2017-01-01 – 2026-04-30` appended to existing 1999-2016 data.

### Gap-filling
Same-day-of-year (DOY) random sampling from the **combined 1999-2026 pool** using
`numpy.random.default_rng(seed=42)`. Falls back to same-month pool for Feb 29
(insufficient DOY pool). Method is identical to the original gap-filling procedure.

### MK station proxy
Station MK was destroyed by a hurricane and has only ~8.1% data coverage post-2017.
A regression-based proxy was constructed using JK (nearest suitable station):

- Training period: 1999-2016 (co-located observations)
- All-day regression: `MK = 0.6593 * JK + 0.0904`  R²=0.441  RMSE=0.784 cm/day
- Wet-day regression (both > 0.05 cm/day, n=1193): `MK = 0.6818 * JK + 0.4089`  R²=0.431
- Applied regression: all-day equation to post-2017 JK values; result clipped ≥ 0
- Training-period bias: **+0.0%** (proxy annual mean = observed annual mean)
- Post-2017 MK: 275 observed | 3,086 JK-regression | 46 zero-fallback (JK also missing)

### Quality statistics (2017-2026 vs 1999-2016)
- All 17 stations within ±25% of historical annual mean (largest delta: LR −11.8%)
- General slight drying trend (−1% to −12%) consistent with regional climate signals
- August −28%, November +77% — real climate signals (active hurricane seasons)
- 2026 calendar-year sanity flag is artifact of 4-month partial year (Jan–Apr only)

`BAM/etc/update_rain_data.py`  |  `--dry-run` flag for pre-write review

## 2. ET/PET — `PET_1999-9-1_2026-4-30.csv`

### Original file
`data/ET/PET_1999-9-1_2016-12-31.csv`
Single column (PET, mm/day), daily, 6,332 rows.

### Original data source
GOES satellite-derived PET (Jacobs et al. 2008 method; see `doc/data/GOES_EDEN_PET_Jacobs_2008.pdf`).
Values closely match the TAYLORRIVER station in the TEF04 MySQL `hydrology` database
(verified match to 3 decimal places, e.g., 1999-09-01: DB=6.379, file=6.38 mm/day).

### DB coverage limitation
The TEF04 MySQL PET data (datatype=`PET`) only extends to **2010-12-31**. The
2011–2016 portion of the existing file was generated by same-DOY climatological
sampling — confirmed by the smooth monthly climatology transition at 2010/2011
(all months within ±14%, most within ±4%).

### Extension method
Same-DOY random sampling from the **1999-2016 record** using
`numpy.random.default_rng(seed=42)`. Falls back to same-month pool for Feb 29.

### Quality statistics (2017-2026 vs 1999-2016)
- Annual mean: **+0.0%** delta (4.256 mm/day both periods — purely climatological)
- All monthly deviations within ±2% — pure sampling noise, no systematic bias
- Boundary check: Dec 25–31 2016 = 2.254 mm/day | Jan 1–7 2017 = 2.207 mm/day (no discontinuity)

### Note on PET variability
Florida Bay PET is dominated by solar radiation and has very low inter-annual
variability. Climatological extension is appropriate and consistent with how
the 2011–2016 portion of the original file was generated.

## 3. Tidal Boundary Conditions — `data/Tide/2000_2026/`

### Original files
`data/Tide/2000_2016/` — 9 files (7 real stations + Region2 + Region3), hourly,
1999-09-01 to 2016-12-31 (~152,000 rows each). Format:
```
Time, WL.(m).demeaned
1999-09-01 12:00 AM EDT, -0.32
```

### Original data source — XTide
Original files were generated by `BAM/etc/CreateTideData.py` using **XTide**
(`tide` command-line tool), which computes tidal predictions from the NOAA/NOS
harmonic constituents database (distributed as `.tcd` files). XTide applies a sum
of cosine terms for each constituent (standard harmonic tidal analysis). The mean
of each prediction series was computed over the generation period and subtracted
("demeaned"). This is **harmonic tidal prediction, not observed water levels**.

### NOAA station identification
All 7 XTide station names map exactly to NOAA subordinate harmonic stations
in the NOAA CO-OPS metadata database (confirmed via API query):

| BAM file | XTide station name | NOAA station ID |
|---|---|---|
| Cape_Sable | Cape Sable, East Cape | TEC4165 |
| Long_Key | Long Key, western end | 8723899 |
| Lignumvitae_Key | Lignumvitae Key, NE side, Florida Bay | 8723824 |
| Snake_Creek | Snake Creek, Hwy. 1 bridge, Windley Key | 8723787 |
| Tavernier_Creek | Tavernier Creek, Hwy. 1 bridge, Hawk Channel | 8723748 |
| Garden_Cove | Garden Cove, Key Largo | 8723622 |
| Little_Card_Sound_bridge | Little Card Sound bridge | 8723534 |

Note: These are NOAA subordinate/harmonic survey stations, not permanent active CO-OPS
gauges. The NOAA CO-OPS predictions API serves hourly predictions for only one
(Little Card Sound, 8723534). XTide's harmonics database includes full constituent
sets for all of them. Cape Sable (TEC4165) uses a non-standard ID prefix; it does
not correspond to a permanent NOAA CO-OPS gauge.

### Why XTide was not re-run
- XTide is a Linux/Unix tool (Ubuntu: `apt-get install xtide`)
- Not available in RedHat `dnf` repositories (TEF04 is RedHat on Azure)
- No Windows build readily available
- Alternative chosen: Python `utide` package (see below)

### Extension method — utide
The `utide` Python package (v0.3.1, compatible with Python 3.9 and 3.13) is used to:
1. **Analyze** the existing 1999–2016 hourly data to extract harmonic constituents
2. **Predict** 2017-01-01 to 2026-04-30 using those constituents

Since the existing data was itself generated from NOAA harmonic constituents (via
XTide), the utide analysis recovers essentially the same constituents. Astronomical
tidal prediction is deterministic; no external data needed.

**Key technical requirement:** Time input to `utide.solve()` must be numpy
`datetime64` arrays (not float days), to ensure correct nodal/satellite corrections
without specifying an epoch manually.

**Validation (Long Key holdout test):**
- Fit 1999–2013 → predict 2014–2016 holdout
- 68 constituents fitted
- Dominant: M2 (A=0.164 m), O1 (0.065 m), K1 (0.064 m), SA (0.046 m), S2 (0.038 m)
- Holdout RMSE: **0.019 m** | R²=0.983

**Per-station fit statistics (full 1999–2016 training, utide v0.3.1):**

| Station | Top constituent (A, m) | In-sample RMSE (m) | 2017–2026 std (m) | 2017–2026 range (m) |
|---|---|---|---|---|
| Cape_Sable | M2 0.385 | 0.044 | 0.354 | [−1.019, +0.892] |
| Long_Key | M2 0.164 | 0.019 | 0.147 | [−0.305, +0.464] |
| Lignumvitae_Key | M2 0.093 | 0.011 | 0.094 | [−0.277, +0.289] |
| Snake_Creek | M2 0.152 | 0.015 | 0.121 | [−0.310, +0.307] |
| Tavernier_Creek | M2 0.191 | 0.021 | 0.164 | [−0.503, +0.391] |
| Garden_Cove | M2 0.310 | 0.031 | 0.249 | [−0.653, +0.622] |
| Little_Card_Sound_bridge | M2 0.077 | 0.009 | 0.066 | [−0.201, +0.156] |

All stations fitted 68 constituents. In-sample RMSE is consistent with the known tidal
character of each site (Cape Sable highest — largest Gulf-side range; Little Card Sound
lowest — sheltered interior micro-tidal).

**DST note:** Original XTide files used local EST/EDT timestamps. Each file contains 18
fall-back duplicate timestamps and ~17 spring-forward gaps, producing 151,969 rows
(vs. idealized 6,332 × 24 = 151,968). The new UTC-like prediction period
(2017-01-01 to 2026-04-30) is gap-free. The combined files each have 233,737 rows;
all 9 files are row-count-consistent so BAM reads them in sync.

### Region2 and Region3 (synthetic derivation)
Per `BAM/etc/CreateRegion.2.3.tide.R`:

**Region2** represents the Gulf interior — attenuated (2/3 amplitude) and 3-hour
delayed version of Cape Sable:
```
Region2[t] = Cape_Sable[t + 3h] * (2/3)
```
Implementation: Cape Sable is predicted 3 hours past the end date
(to 2026-05-01 02:00) to fill the last 3 rows.

**Region3** represents the Atlantic interior — amplified (4/3 amplitude) and
3-hour advanced version of Long Key:
```
Region3[t] = Long_Key[t - 3h] * (4/3)
```
Implementation: Existing 2016-12-31 21:00–23:00 Long Key values provide the seam
for the first 3 hours of 2017.

### New boundary configuration
`data/Boundary/Basin_Tide_Boundary_2000_2026.csv`
(replaces `Basin_Tide_Boundary_2000_2016.csv` for the extended POR run)

### Output files
`data/Tide/2000_2026/` — 9 files, 233,737 rows × 7.1 MB each:
`Cape_Sable`, `Long_Key`, `Lignumvitae_Key`, `Snake_Creek`, `Tavernier_Creek`,
`Garden_Cove`, `Little_Card_Sound_bridge`, `Region2`, `Region3`
(all named `<station>_1999-09-01_2026-04-30.csv`)

`data/Boundary/Basin_Tide_Boundary_2000_2026.csv` — basin→file mapping for
basins 59–68 (replaces `Basin_Tide_Boundary_2000_2016.csv`)

## 4. EDEN Runoff Stage

### Original file
`data/Runoff/EDEN_Stage_OffsetMSL.csv`
8 stations (S15–S22), daily stage (m, MSL offset), 6,513 rows,
1999-09-01 to **2017-06-30**.
Columns: `Date, S22, S21, S20, S19, S18, S17, S16, S15`

### What S15–S22 are
These are **EDEN water surface model grid output points**, not physical
monitoring stations. Dr. Joseph Park (NPS) selected 8 specific UTM Zone 17N
coordinates across the southern Everglades/Florida Bay boundary as representative
locations for estimating hydraulic head and freshwater runoff into the coastal
basins (streamflow, sheetflow, groundwater).

Coordinates (from `BAM/etc/FBM_EDEN.xyLocator.txt`):

| Station | UTM Easting (m) | UTM Northing (m) | Lat (°N) | Lon (°W) |
|---|---|---|---|---|
| S15 | 520,000 | 2,794,000 | 25.2622 | 80.8014 |
| S16 | 525,000 | 2,792,500 | 25.2486 | 80.7518 |
| S17 | 530,000 | 2,790,000 | 25.2259 | 80.7022 |
| S18 | 535,000 | 2,791,500 | 25.2394 | 80.6525 |
| S19 | 540,000 | 2,794,500 | 25.2663 | 80.6027 |
| S20 | 545,000 | 2,795,000 | 25.2707 | 80.5531 |
| S21 | 550,000 | 2,795,500 | 25.2751 | 80.5034 |
| S22 | 555,200 | 2,795,700 | 25.2767 | 80.4517 |

### Data source
**USGS Everglades Depth Estimation Network (EDEN) v3 water surface model**
Reference: Telis et al. (2015), USGS SIR 2014-5209; `http://sofia.usgs.gov/eden/`

EDEN produces daily interpolated water-surface elevations (cm, NAVD88) on a
400 m UTM Zone 17N grid covering the greater Everglades. NetCDF files are served
via the USGS South Florida THREDDS/OPeNDAP server:
`http://sflthredds.er.usgs.gov/thredds/dodsC/eden/surfaces/YYYY_qN.nc`

The server holds all quarters from 1991 Q1 through the present. The NetCDF
structure is: `stage[time × y(405) × x(287)]` (Float32, cm NAVD88),
with 1-D coordinate arrays `x` and `y` in UTM metres (400 m grid spacing,
SW corner at x=463,400 m, y=2,790,200 m).

**API investigation result:** S15–S22 are model output grid points, not NWIS
gage stations. No coordinate-query API or THREDDS OPeNDAP subsetting was needed
beyond nearest-grid-cell selection; NWIS, ScienceBase, and the EDEN gage
station list all confirmed these points are not physical gauges.

### MSL offset
The R script (`EDEN_Stage.R`) applies: `output_m = NAVD88_m − MSL_NAVD88`
where `MSL_NAVD88 = −0.148 m`. Therefore:
`output_m = NAVD88_m + 0.148 m` (shifts from NAVD88 datum to MSL anomaly)

### Extension method
OPeNDAP queries to the THREDDS server — no full-grid file downloads.
For each of 36 quarterly files (2017 Q3 through 2026 Q2):
1. Open dataset lazily (only DDS metadata transferred)
2. Load 1-D x and y coordinate arrays (~5 KB total)
3. Find nearest grid cell index for each of the 8 stations
4. Fetch each cell's time series via `isel()` — one constrained OPeNDAP
   request per station per quarter (~3 KB per cell)
5. Convert cm → m, apply MSL offset, round to 3 decimal places
6. Trim 2026 Q2 to 2026-04-30

### Quality statistics (2017–2026 vs 1999–2017)
| Station | 1999–2017 mean (m) | 2017–2026 mean (m) | Delta |
|---|---|---|---|
| S22 | +0.168 | +0.214 | +27.6% |
| S21 | +0.185 | +0.266 | +43.2% |
| S20 | +0.097 | +0.177 | +82.7% |
| S19 | +0.090 | +0.156 | +73.9% |
| S18 | +0.088 | +0.143 | +63.2% |
| S17 | +0.082 | +0.168 | +106.3% |
| S16 | +0.103 | +0.228 | +122.1% |
| S15 | +0.052 | +0.125 | +138.8% |

Large percentage increases reflect small absolute baselines (52–185 mm) and
a genuinely wetter 2017–2026 period (active hurricane seasons, increased CERP
water deliveries to the Slough). Absolute stage increases are 46–90 mm — within
observed inter-annual variability. Seam at 2017-07-01: all stations within

### Data source
**TEF04 MySQL** `hydrology.measurement`, `datatype='salinity'`
Units: ppt (practical salinity units; ppt ≈ PSU in this range)
Resolution: sub-daily (continuous monitoring); aggregated to daily mean

post-2017 included values of 46,354 and 231,597 ppt from sensor fouling/air exposure).

**Gulf_1 and Ocean_1** (open-ocean boundary columns) are not in the TEF04 DB.
These are gap-filled entirely from the existing 1999–2016 record using the same
same-DOY random sampling method (see Gap-filling below).

18 of 20 DB stations: >99% daily coverage.
MK: 85.3% — partial data loss from Hurricane Irma (2017); the station continues
but with intermittent gaps. 500 days gap-filled.
MB, MD: 92–98% — offshore mooring stations with periodic maintenance gaps.

### Gap-filling
Same-day-of-year (DOY) random sampling from the **combined 1999–2026 pool**,
`numpy.random.default_rng(seed=42)`. Falls back to same-month pool for Feb 29.
Applied to all 22 columns; 0 NaN values remain after gap-fill.

## 6. Stage (observed / validation)

### Original file
`data/Stage/DailyStage_1999-9-1_2016-12-31.csv`
19 stations, daily m (MSL offset), 6,332 rows, 1999-09-01 to 2016-12-31.
Contains 3,578 NaN cells (pre-existing; not backfilled).
Columns: BA, BK, BN, BS, DK, GB, HC, JK, LB, LM, LR, LS, MK, PK, TC, TR, WB, MB, MD

### Data source
**TEF04 MySQL** `hydrology.measurement`, `datatype='stage'`
Units in DB: **feet, NAVD88 datum**
Units in BAM file: **metres, MSL-offset** (same convention as EDEN runoff stage)

### Unit / datum conversion
```
output_m = DB_ft × 0.3048 + 0.148
```
Derivation: `output_m = NAVD88_m − MSL_NAVD88 = NAVD88_m − (−0.148) = NAVD88_m + 0.148`
where `MSL_NAVD88 = −0.148 m` (MSL is 0.148 m below the NAVD88 reference, consistent
with the EDEN runoff stage conversion).

Cross-check (BA, 2016-12-25): DB −0.596 ft → −0.034 m; BAM file = −0.011 m;
residual = 0.023 m. Similar residuals (~0.015–0.025 m) at all stations reflect
station-specific MSL variations across the Bay — within acceptable model tolerance.

**Sentinel / plausibility filter:** Sub-daily readings outside −15 to +15 ft are
excluded before averaging. This removes −99999 and −99.99 ft sentinel values (BN:
1,394 rows; BA, LB: ~1–3 rows) and an obvious 299.8 ft spike at DK, while
preserving the full Florida Bay range including Hurricane Irma surge (~13 ft peak).

### Station notes
- **MK**: Stage station permanently destroyed by Hurricane Irma (2017-09-10).
  Last DB record: 2017-09-23. Only 7.4% coverage in extension period (Jan–Sep 2017
  pre-storm data). The remaining 3,154 days (2017-09-24 to 2026-04-30) are
  gap-filled by same-DOY random sampling from the combined pool.
  *Note:* MK shows a +0.493 m seam jump at 2017-01-01; the Jan–Sep 2017 DB data
  reads ~+0.45 m, possibly reflecting a datum transition at MK in early 2017.
  Given that MK is 92.6% gap-filled and likely used for validation rather than
  forcing, this has limited practical impact.
- **MB, MD**: Offshore stations; 92–97% coverage. 267 and 89 days gap-filled.
- **BN**: 164 days gap-filled after removing −99999 sentinel rows.

### Gap-filling
Same-DOY random sampling from the combined 1999–2026 pool, seed=42,
applied **only to the extension period** (2017-01-01 to 2026-04-30).
Pre-existing NaN values in the 1999–2016 portion are preserved unchanged (3,578 cells).
0 NaN values remain in the extension period.

### Quality statistics (seam at 2017-01-01)
Most stations show seam deltas ≤ 0.095 m — consistent with a seasonal dry-season
boundary. Annual means for 2017–2026 range +0.004 to +0.105 m MSL offset,
plausibly higher than the 1999–2016 baseline (+0.001 to +0.036 m) reflecting
wetter recent conditions. All annual station-means within plausible range
(−1.0 to +1.5 m). Percentage deltas appear extreme (+1000–67000%) solely because
the 1999–2016 means are near zero; absolute shifts are 0.02–0.10 m.

## 8. S197 Flow

Single column `cfs.daily`, 6,630 rows, 1999-09-01 to 2017-10-25.
Note: despite the column name, values are stored in **kcfs (cfs ÷ 1000)**.

### Data source
**TEF04 MySQL** `hydrology.measurement`, `station='S197'`, `datatype='flow'`
Units in DB: cfs (cubic feet per second), daily resolution (one row per day).
S197 is the Taylor Slough Boatlanding water control structure, ENP.
Flow is controlled/gated; ~90th percentile = 0 (gate closed most days).

**Primary source:** `datatype='flow'` (daily, 1970–present)
**Fallback:** `datatype='flow_realTime'` (sub-daily ~10 min, 2012–present),

### Data gap (2018-10-17 to 2020-06-09)
The `flow` datatype has null `measurement_value` for 602 consecutive days in this
window. The `flow_realTime` datatype is also mostly null for this period (only 2
days recovered via fallback). This appears to be a period when the structure was
not in service or data was not logged to the daily record.

**Gap treatment:** Same-DOY random sampling from the combined 1999–2026 historical
pool, seed=42. Since S197 flow is 0 on ~90% of days, the gap-fill correctly
produces mostly-zero values with occasional wet-season pulses drawn from the
historical distribution.

**Overlap note:** 4 days in October 2017 (post-Hurricane Irma) show differences
of 0.12–0.47 kcfs between the existing BAM file and the current DB values. These
are attributed to post-Irma data revisions in the DB and do not indicate a unit

## Config Update (final step)

After all data files are extended, update `config.py` (or equivalent BAM argument
defaults) to reflect the new period of record:
```python
BAM_DATA_START = '1999-09-01'
BAM_DATA_END   = '2026-04-30'
```
Also update the dashboard date picker bounds in `app.py`.
